# 论文主线调整建议：从全量联邦扩散训练转向联邦场景下的轻量化安全更新

## 建议后的核心表述

论文不再把重点放在“从零开始全量训练一个联邦扩散模型”，而是转向：

**在联邦场景下，研究扩散模型的轻量化训练、低开销参数上传，以及可验证的安全更新链路。**

更贴合题目的中文表达可以写成：

> 本课题以 latent diffusion 为基础生成框架，围绕联邦环境下的大模型训练代价高、更新传输成本大、跨端更新难以验证等问题，提出一种面向 UNet-LoRA 的轻量化联邦训练与安全更新原型。系统冻结 VAE 与文本编码器，只对 UNet attention 层中的 LoRA 适配器进行本地训练和联邦聚合，并结合可逆隐写封装与更新审计机制，对客户端上传更新的完整性进行验证。

## 为什么这条主线更合理

- 扩散模型的大头不只在参数量，还在训练时的梯度、优化器状态和激活开销。
- 在联邦环境下，全量同步整个 UNet 既昂贵，也很难稳定落地。
- LoRA 的价值在于：冻结原始预训练权重，只训练低秩适配器，因此客户端上传的是小得多的增量参数，而不是整个扩散主干。
- 这样可以把论文的核心问题更聚焦为三件事：
  - 联邦训练是否还能保持可用的生成效果；
  - LoRA 聚合是否能在通信成本受控的情况下工作；
  - 上传更新是否能够被可逆封装与审计验证。

## 当前原型系统的准确口径

- 训练对象：`UNet-LoRA`
- 冻结模块：`VAE`、`Text Encoder`
- 聚合对象：LoRA 参数矩阵
- 当前聚合方式：按客户端样本量加权平均
- 当前审计含义：验证每个客户端上传的 LoRA 更新在封装、恢复后是否保持完整一致
- 当前系统证明的是：**更新传输完整性**
- 当前系统尚不证明的是：**严格形式化的强安全聚合证明**

因此论文里最稳妥的写法应是：

> 系统当前验证的是联邦更新在封装与恢复链路中的端到端完整性，而不是对强对手模型下机密性与抗攻击性的完整形式化证明。

## 本地已跑通的代表性实验

### 1. STL10 联邦 LoRA 审计版

- 运行目录：`outputs/tiny_sd_stl10_lora_fed_audit_local_quick_run`
- 设置：`4` 客户端，`每轮选 2`，`8` 轮，`LoRA rank=16`
- 最优损失：`0.172751`
- 审计结果：`16/16` 全部验证通过
- 平均压缩后负载：约 `6,958,194` bytes / client / round

这组实验适合说明：

- LoRA 联邦训练是可跑通的；
- 更新封装与恢复链路能稳定工作；
- 引入审计后，训练过程仍保持数值稳定，没有出现 `NaN`。

### 2. STL10 中心化 LoRA 对照

- 运行目录：`outputs/tiny_sd_stl10_lora_central_local_quick_run`
- 设置：`1` 客户端，`1` 客户端/轮，`8` 轮，`LoRA rank=16`
- 最优损失：`0.165078`
- 审计记录：`0`

这组实验适合说明：

- 中心化条件下优化更直接，损失略低于联邦版；
- 但它不涉及跨客户端更新传输，因此也不具备联邦审计链路的展示意义。

### 3. STL10 联邦全客户端聚合版

- 运行目录：`outputs/tiny_sd_stl10_lora_fullclients_local_quick_run`
- 设置：`4` 客户端，`每轮 4 个全参与`，`8` 轮，`LoRA rank=16`
- 最优损失：`0.158269`
- 审计结果：`32/32` 全部验证通过

这组实验适合说明：

- 更高的客户端覆盖度可以带来更好的短期优化结果；
- 但代价是每轮上传更多更新，审计与传输负载同步增加。

### 4. STL10 联邦低秩容量消融（rank=4）

- 运行目录：`outputs/tiny_sd_stl10_lora_rank4_local_quick_run`
- 设置：`4` 客户端，`每轮选 2`，`8` 轮，`LoRA rank=4`
- 最优损失：`0.192928`
- 审计结果：`16/16` 全部验证通过
- 平均压缩后负载：约 `1,777,222` bytes / client / round

这组实验适合说明：

- 更小的 LoRA rank 会显著降低通信负载；
- 但过低的低秩容量会带来性能回落；
- 这正好体现“轻量化训练”和“生成能力”之间的权衡。

### 5. Fashion-MNIST 联邦泛化实验

- 运行目录：`outputs/tiny_sd_fashion_lora_fed_local_quick_run`
- 设置：`4` 客户端，`每轮选 2`，`8` 轮，`LoRA rank=16`
- 最优损失：`0.146743`
- 审计结果：`16/16` 全部验证通过

这组实验适合说明：

- 轻量联邦 LoRA 训练不局限于 STL10；
- 在更简单的数据域上也能保持稳定优化与审计验证。

### 6. MNIST 联邦泛化实验

- 运行目录：`outputs/tiny_sd_mnist_lora_fed_local_quick_run`
- 设置：`4` 客户端，`每轮选 2`，`8` 轮，`LoRA rank=16`
- 最优损失：`0.098984`
- 审计结果：`16/16` 全部验证通过

这组实验适合说明：

- 在简单结构数据上，LoRA 联邦更新可以更快收敛；
- 这为论文里的“跨数据集可迁移性”提供了初步证据。

## 当前最适合放进论文的指标

建议保留四类指标：

### A. 联邦训练效果

- `avg_client_loss`
- 最优轮 `best_loss`
- 最终轮 `last_round.avg_client_loss`

用途：

- 对比中心化与联邦训练是否保持稳定可训练；
- 对比不同数据集、不同聚合方式的优化难度。

### B. LoRA 聚合效果

- `lora_numel`
- `mean_client_update_l2`
- `aggregation_delta_l2`
- `global_lora_l2`

用途：

- 说明 LoRA 更新规模和聚合变化幅度；
- 量化“联邦每轮到底在上传和聚合什么”。

### C. 审计与传输效果

- `audit_records`
- `audit_verified`
- `payload_bytes_raw_mean`
- `payload_bytes_compressed_mean`

用途：

- 说明每轮上传更新是否被完整恢复；
- 说明安全封装链路带来的实际传输规模。

### D. 消融实验指标

- 中心化 vs 联邦：比较 `best_loss`
- `rank=16` vs `rank=4`：比较性能与负载
- `clients_per_round=2` vs `clients_per_round=4`：比较覆盖度与代价

## 当前最推荐的论文比较故事

### 主线 1：联邦引入后训练是否仍可用

- 用 `STL10 中心化` 对照 `STL10 联邦 2 客户端/轮`
- 结论：联邦化带来轻微性能回落，但训练仍然稳定，且获得了联邦更新链路与审计能力

### 主线 2：LoRA 是否真的降低通信成本

- 用 `rank=16` 对照 `rank=4`
- 结论：更低 rank 的确显著减小上传负载，但会牺牲部分效果

### 主线 3：审计链路是否可行

- 用所有联邦实验的 `audit_verified / audit_records`
- 当前结果：已跑通的联邦 LoRA 组全部达到 `100%` 完整性验证通过

## 建议在论文中避免的写法

- 不要写成“本文完成了全量联邦扩散模型训练框架”
- 不要写成“本文证明了 secure aggregation 的完备安全性”
- 不要把 LoRA 更新完整性验证写成“机密性证明”

## 建议采用的写法

- “本文聚焦联邦场景下扩散模型的轻量化训练与安全更新”
- “本文在冻结大模型主体参数的前提下，对 UNet-LoRA 更新进行联邦聚合”
- “本文原型验证了联邦更新在隐写封装与恢复链路中的完整性”
- “本文展示了生成效果、通信负载与审计可验证性之间的系统性权衡”
